Interactions not shown above

Interactions available but not illustrated in these examples:
  • Search / filter — typing in the search box filters entities across all categories. Matching is against entity names, descriptions, slot names, and variable labels. Categories with no matches collapse automatically; categories with matches auto-expand.
  • Pinning / unpinning — hover over an entity name to reveal the pin button (○). Click to pin; the entity appears in the Pinned section. Click the 📌 on a pinned entity to unpin. Pin state persists across sessions (localStorage).
  • Hovering a range count badge — hovering a range count (e.g., the 5 enum count on Condition) shows a tooltip listing the range names (e.g., "ConditionConceptEnum, ConditionSeverityEnum, …").
  • Clicking a range count badge — opens the slot drilldown and scrolls to / highlights the slots with that range type. For example, clicking Condition's 5 opens the slot drilldown with the 5 enum-ranged slots highlighted. Primitive-type slots are shown first, so clicking the type count just opens the drilldown at the top.
  • Clicking a range badge in the slot table — shown in Examples 4 and 5. Enum-range badges expand an inline enum detail card (permissible values, other usages). Class-range badges expand an inline entity summary card (slots, description, "referenced by" list) with a "Go to entity" link to navigate away if desired.
  • Column visibility — the Show toggles (Slots, Ranges, Vars, Desc) in the controls bar hide/show columns across all entity tables. Desc is off by default to save space.
  • Expand all / collapse all — the Expand controls quickly open or close all category groups.
  • Inherited vs. own slots — in the slot drilldown, inherited slots are tagged with their source (e.g., Entity, Observation). Overridden slots show overrides Observation. A future toggle could hide inherited slots to show only the entity's own definitions.
  • Dark mode — toggle via the button in the controls bar. Try it on this page!

Open design questions

  • Where do links fit? The inline entity summary cards (Example 5) and "Referenced by" lists provide the same relationship information that links visualize, but in a more compact, contextual way. Links could still work as an optional overlay or a separate "Relationships" tab/view for people who want the big-picture graph.
  • Category groupings — the categories shown here (Admin/Study, Clinical, Lab/Biospecimen, Measurement/Observation, Survey, Files/Other) are a first draft. Entities can appear in multiple categories (e.g., Condition appears in both Pinned and Clinical). Should categories be user-configurable?
  • Variable column adaptation — should variable tables automatically adjust their columns based on what data is present (hide Unit/Description for Condition-type variables), or should all variable tables be uniform?
  • Slot drilldown vs. separate panel — drilldown shown here is inline (expands below the row). An alternative is a side panel or modal. Inline keeps context; a panel allows more room. Could make this configurable.
  • Enum detail depth — the inline enum card (Example 4) shows permissible values and which slots use the enum. Enums also have inheritance relationships and other metadata fields we haven't incorporated yet. How much enum detail should be accessible inline vs. requiring a dedicated enum browser?
  • Nested inline cards — what happens when you click a range badge inside an inline card? (E.g., clicking UnitOfMeasurementEnum inside the Quantity card.) Two approaches:
    • Replace: close the current card, open the new one at the slot-table level. The breadcrumb updates to show the full trail and clicking breadcrumb segments goes back. Simpler, avoids deep nesting.
    • Nest: expand the new card inside the current one. Risk of getting visually deep, but could be mitigated by scaling each nesting level down (e.g., font-size: 0.9em per level). Keeps all context visible at once.
    Leaning toward replace with animated breadcrumb update for most cases.
  • Navigation history — the current app encodes state in the URL so the browser back button works (though it's buggy). For the new design, the key states to encode would be: which entity is expanded, which drilldown tab, and which inline card is open. A small parameter set.
